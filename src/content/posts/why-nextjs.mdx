## 前情提要

之前我用 [Astro](https://astro.build) 搭过一版个人博客，
基于 Mizuki/Fuwari 类的二次元主题。Astro 的优点很明显：

- **静态优先**：默认生成纯静态站点，CDN 命中率近乎 100%
- **岛屿架构**：只在需要交互的地方加载 JS，首屏体积极小
- **生态轻量**：学习曲线短，跑起来快

那为什么还要换？因为我想做的功能远远超出"展示内容"。

## 触发迁移的功能清单

| 功能 | 静态站点能做吗？ | 备注 |
| --- | --- | --- |
| 文章 + RSS | ✅ | 都能 |
| Giscus 第三方评论 | ✅ | 都能 |
| **自建评论 + 数据库** | ❌ | 需要后端 |
| **浏览量 / 点赞** | △ | 静态需要外部 API |
| **私有管理后台** | ❌ | 需要后端 + 身份验证 |
| **邮件订阅 (Newsletter)** | △ | 静态需要嵌入第三方表单 |
| **AI 问答 / 摘要** | ❌ | 需要 API 路由 |
| **付费墙 / 会员** | ❌ | 需要后端 |
| **多语言** | ✅ | 都能 |

打勾的部分两者都能做，但**所有"❌"的功能都需要后端**。
Astro 本身也支持 SSR adapter，但生态远不如 Next.js 丰富——
NextAuth、Drizzle、各种 UI 组件库都首先适配 Next.js。

## 决策

最终选择：**Next.js 16 + Postgres + Vercel**。

代价：

- 复杂度更高（学习曲线陡）
- 部署需要 serverless 而不是纯静态
- 首屏 JS 比 Astro 多

但换来的是**功能上限基本没有天花板**——这正是我想要的。

> 如果你只想做一个内容展示的博客，**Astro 仍然是更好的选择**。
> 但如果你想做一个"个人 SaaS"风格的网站，Next.js 更合适。
